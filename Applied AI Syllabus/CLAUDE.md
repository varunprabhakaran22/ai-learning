# AI Learning Notes — Project Guide

This project is a self-directed AI/LLM engineering curriculum (see `Applied AI Syllabus.md`). Notes are organized in `Day N/` folders.

**Scope:** everything below applies only to `Day N/` folders. The `Claude Code in Action/` folder has its own `CLAUDE.md` with separate rules (it's a different curriculum — the Anthropic Skilljar course, not the `Applied AI Syllabus.md` one). The `Python/` folder is also a separate, independently-paced curriculum (Python fundamentals for AI/ML) with its own `CLAUDE.md` — it does not follow the `Week_N` day structure or pacing.

## Structure per day
- `Theory.md` — the core mental model for the day's topic: mechanics/facts the user must know and will keep referencing. Self-contained, no external source needed to make sense of it. If a concept is "core" (would show up in Recap as a must-know), it belongs here — regardless of how deep/technical it is. **Any new syntax, API field, rule, or definition introduced while writing Theory.md must be explicitly covered in Theory.md itself** — never left implicit or assumed as "obvious," even if it seems minor (e.g. what `stop_reason` is, where it comes from, why it can't be hallucinated). If it's not written down, the user will end up guessing at it instead of knowing it.
  - **Always go one level under the mechanism, not just to it.** A mock interview (2026-07-08, covering Weeks 1–3) surfaced a recurring gap: the user's answers consistently land on the right *behavior/direction* (what to do, that something is a trade-off) but stop before the *underlying mechanism* (why it happens at the token/API/architecture level) and before *concrete specifics* (a named term, an exact field, a number). Theory.md must not stop at "X is a trade-off" or "the model decides based on Y" — it must state the actual mechanism underneath: e.g. not just "temperature=0 is deterministic" but why it can still vary (greedy decoding + floating-point non-associativity in batched GPU execution); not just "structured output enforces JSON" but how (constrained decoding masks invalid tokens at the sampling layer); not just "tool descriptions matter" but naming the actual failure class when they don't (e.g. SSRF, not just "bad description"). When writing or revising Theory.md, ask "if this exact concept were probed one level deeper in an interview, is the next-level answer already written down here?" — if not, add it.
  - This mechanism-depth requirement applies to Theory.md content itself, going forward — it is separate from and does not apply to interview-style Q&A gotchas (those get captured live during mock interviews, not pre-written into Theory).
  - **Build up from true zero before adding mechanism depth — never open on the deep mechanism itself.** RAG & Knowledge Systems Day 1 — Embeddings & Vector Search's Theory.md originally opened directly on "an embedding is a vector from a trained transformer, via contrastive learning" — technically correct and appropriately deep, but the user couldn't follow it because nothing established what a vector even IS first. Rewritten with a ⓪/① foundation: ⓪ states the plain-language problem being solved (LLMs don't know your data, so find-and-paste relevant pieces — RAG), ① builds the base concept with a trivial, concrete, non-AI example (a vector is just `[3, 7]`, a point on a 2D graph — close points are similar, far points aren't) BEFORE the neural-net/training explanation is introduced. Every subsequent fact should be reachable by someone who only just read the fact before it — no forward-referenced jargon, no assumed background. This does NOT mean cutting depth or simplifying the mechanism itself (contrastive learning, HNSW graph internals, the cosine formula all stayed, in full) — it means sequencing so each layer has ground under it before the next layer loads on top. When writing or revising Theory.md, check each section in order: could someone who just read everything above this point, and nothing else, follow this next part? If a section's first sentence requires knowledge not yet established earlier in the same file, add the missing foundation before it, don't just push through.
  - **No forward-referencing a term to a later day as a substitute for explaining it now.** RAG & Knowledge Systems Day 1 — Embeddings & Vector Search's Theory.md mentioned "chunking" (as the reason whole-document embeddings lose precision) but never explained what chunking actually is or how it works, leaving it dangling until Day 2 — the user caught this as an unclosed gap during Day 1's own recap. If Theory.md introduces a term/concept by name to support an argument, it must explain that term to at least a basic working level IN THAT SAME Theory.md, even if the term gets its own full/deeper treatment on a later day. "Covered in depth on Day N" is fine for expanding on a term; it is never a valid reason to leave the term itself unexplained today. Before finishing any Theory.md, scan every named term/concept introduced and confirm each one either (a) is explained inline, or (b) was already explained earlier in the same file — never (c) left as a bare name pointing forward.
  - **Target bar, set after RAG & Knowledge Systems Day 1 — Embeddings & Vector Search's back-and-forth actually closed every gap: every Theory.md going forward must be that dense, assume ZERO prior knowledge (treat the user as an absolute beginner on that specific concept, even if they're experienced elsewhere), and still fully cover the topic to the depth a senior AI engineer (interviewing for senior/staff-level roles) needs — not a simplified/intro version that gets "topped up" later. Depth and beginner-friendliness are NOT a trade-off here: start from true zero (per the "build up from true zero" rule above) but do not stop until the topic is covered at full professional depth. If reaching that depth requires more sections/length than a typical day's Theory.md, that's expected and fine — do not compress for brevity at the cost of completeness.**
  - **Each Theory.md must read as one continuous flow, not a set of jump-around facts.** Never write as if the reader might already know a concept ("if you're familiar with X..." / "as you may know..." / assuming a skip is safe) — always state it as if this is the first time they're encountering it, in strict dependency order, so reading top-to-bottom in order is the only path and never requires jumping ahead or already knowing something not yet introduced in that same file. This applies within a single Theory.md; carrying a truly foundational concept forward from an EARLIER day's Theory.md (already read and recapped) is fine — the ban is on assuming knowledge that hasn't been explicitly built up somewhere the user has already read.
  - **No single mandatory section-shape — structure follows what the topic needs.** RAG & Knowledge Systems Day 1's ⓪①②③④⑤ numbered build-up (problem first, simplest-possible core idea, then layering in the real mechanism one dependency at a time, ending in a named trap/gotcha section and a compressed summary) is one example of a structure that worked well — not a mold every day must be forced into. Day 3 (Advanced Retrieval Strategies) is an equally good but differently-shaped example, structured around comparing several named techniques side by side instead of one single chain. Pick whichever shape actually fits the day's topic: a single-core-mechanism topic wants the ⓪-⑤ chain; several named alternatives that need telling apart want a compare-and-contrast layout; a decision framework wants a decision-tree/flowchart shape. What's required regardless of shape: a problem-first opening, ground-up sequencing, explicit formulas/mechanisms (not named-but-unexplained concepts), at least one concrete worked example per major idea, named failure modes/gotchas, and a compressed summary at the end.
  - **Completeness in the first draft is not optional — the doubt-and-revise loop cannot catch an omission, only an unclarity.** The workflow (write Theory.md → user reads and raises doubts → revise) only works for things that are present but confusing; it has no mechanism to surface a definition, mechanism, or piece of architecture that the first draft never mentioned at all — there's nothing on the page to prompt a doubt about it, so a silent gap stays silent permanently (or resurfaces only by accident, as a bare forward-reference on some later day). Before treating any first-draft Theory.md as done, actively check: does this topic have any definition, named technique, architectural piece, or formula that this draft doesn't expose anywhere, even briefly? If so, add it now — "the user can ask about it later" is not a valid reason to omit something genuinely in scope for the day, because omission removes the only chance to catch the gap.
- `Lessons.md` — notes tied to a specific external source (an article, paper, video) or hands-on experiments. Always name the source at the top of each section (e.g. "Part A — The Illustrated Transformer (Jay Alammar)"). If a section has no external source and no experiment — it's core theory, and belongs in Theory.md instead.
- `Recap.md` — the user's own minimal summary of what they understood, written **after** reading Theory/Lessons. Structure it generically by topic first — as if summarizing Theory.md/Lessons.md cold, in the same section shape those files use (a short section per concept/technique) — key facts/rules only, no long explanations, no dumping source content. Where the conversation that day surfaced a correction, a worked example, or a detail not obvious from Theory.md alone, fold it into that same topic's section (e.g. as a trailing "**From our discussion:**" line), rather than writing a separate conversation-shaped list bolted onto the end. Not every section will have one — only add it where the conversation actually added something beyond what Theory.md already says. Always end with a short "still need to cover" list pointing to what's next.
- **No bare cross-reference pointers — inline the actual point (2026-07-16).** A file's own `## ① ...`/`## ② ...` numbered section headers (structuring that single file's own build-up) are fine as-is. But never point to a numbered section — same-file ("(② above)") or another day's ("Day 1 fact ①") — as a stand-in for its content (e.g. "Original plan: [the 4-step list from ①]"). Write the actual fact inline, in one short clause, at the point of reference. Reason: the reader can't mentally resolve dozens of these pointers across files; the file must read standalone. This does not apply to a file's own `## ①..⑤` headings or its own end-of-file "Summary — ⓪ through ⑤" recap block — those are structural, not pointers, and stay as-is.
- `example.js` — small runnable code example tying together the day's concepts

## Code style for examples
- No classes. Use plain functions and plain objects/config — even if the syllabus showcase task suggests a class-based design (e.g. "Build a PersonaEngine class"). Keep the concept demonstrated, skip the class wrapper.
- Keep examples minimal — just enough to demonstrate the concept, not production-grade.

## Explanations
- Whenever explaining a flow/process (e.g. a request loop, a multi-step mechanic), lead with an ASCII flow diagram before the prose walkthrough — diagrams first makes it easier to follow.
- Every code example (`example.js`, or any `mcp-*.js`/multi-file example) must have an ASCII flow diagram showing its actual runtime flow (who calls what, in what order, what data passes between steps) in its accompanying explanation doc — not just prose walkthroughs.
