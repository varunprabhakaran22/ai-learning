# Day 1 Recap — Agent Architecture Patterns

## Chain vs. agent — the actual dividing line
A chain's control flow lives in fixed code you wrote (step 1 always runs, then step 2, regardless of what step 1 returned) — the LLM only fills in content within a step. An agent's control flow lives in the LLM's own repeated reasoning: it decides whether to call a tool, which one, what to do with the result, and when the goal is actually done — not hard-coded in advance. "Agent" is an architecture (a specific arrangement of components), not a new API feature or model capability.

**From our discussion:** the tool-call mechanism itself never changes between a Week 3 ReAct loop and a full Week 5 agent — planning/memory/executor don't replace tools+ReAct, they wrap around it, deciding when to invoke that same loop and what subtask it's solving right now.

## Pillar 1 — Tools (Action)
Nothing new mechanically — this is Week 3's `ToolRegistry` + request→execute→inject→continue loop, unchanged. Tools are the agent's only way to affect or observe the outside world; without this pillar an agent could only think in text, never act. "Tools" and "Action" are the same pillar, not two separate ones.

**From our discussion:** Perplexity "leaning on the tools pillar" means its core registered tool is basically "search the web" — same registration mechanism as `getHighAUMUsers` in Week 3, just one dominant tool instead of several, with comparatively thin planning/memory around it.

## Pillar 2 — Planning
An extra LLM call (or the same call, prompted differently) that outputs a structured, ordered subtask list BEFORE any tool-calling begins — your code then executes that list in dependency order. Same "model outputs structured text, code parses and acts on it" pattern as tool_use, applied to a whole task instead of one function call. Two named strategies: **linear planning** (one ordered list, executed top to bottom — fine when the task is well-understood upfront) vs **tree-of-thought** (generate several candidate next-steps, evaluate which looks best, continue down that branch, abandon the rest — costs more calls, reserved for genuinely ambiguous reasoning tasks).

Named failure modes: **stale plans** (a rigid plan has no mechanism to notice an early assumption was wrong — fixed by dynamic replanning), **over-decomposition** (breaking a simple task into needlessly many tiny subtasks wastes calls), **under-specified subtasks** (a vague step like "research the competitor" gives the executor nothing concrete to act on).

**From our discussion:** planning is essentially a todo list with dependencies, generated by an LLM instead of typed by hand — what makes it more than a static todo list is that the executor can regenerate/patch it mid-run if a later step reveals an earlier assumption was wrong (a plain todo list never rewrites itself).

## Pillar 3 — Memory
Solves two problems a short Week 3 loop never hit: the context window's hard token limit, and attention dilution from cramming in irrelevant past history. Four types: **in-context (short-term)** — the current conversation, cheapest, bounded by the window, gone when the session ends. **External/vector memory** — Week 4's exact embed→store→cosine-similarity→topK mechanism, repointed at the agent's own past actions/observations instead of documents, so only relevant history gets retrieved instead of all of it. **Episodic memory** — memory of specific past sessions/outcomes, usually just a tagged kind of entry in the same vector store. **Semantic memory** — durable standing facts, stored the same way, distinguished by not being tied to one episode. Episodic/semantic entries get auto-summarized before storing (an LLM condenses a session into a short summary first) rather than storing raw transcripts verbatim — same near-duplicate-flooding trap from Week 4 Day 1, now showing up for the agent's own history.

## Pillar 4 — Executor
The runtime loop tying the other three together: pull next subtask from the plan → retrieve relevant memory → ask the LLM to decide the next action (tool call runs Week 3's exact loop here) → execute → store the outcome into memory → check whether the plan is still valid (replan if not) → repeat or stop. This loop's replan step is the concrete difference from a chain — a chain has no step that notices its own plan is broken. Stopping condition is never left to vibes: production agents combine (a) the LLM's own explicit "done" signal, (b) a hard iteration cap as a backstop independent of the model's judgment (Week 3 Day 3's `maxIterations`, reused), and (c) the plan's own completion state.

**From our discussion:** every individual step is still a structured prompt in, structured text out — same mechanism as Week 1-3. What makes an agent different from "one very structured prompt" isn't the mechanism of a single call, it's that later prompts get built using the actual results of earlier calls across multiple separate calls over time — content only knowable after the previous step ran, not fixed in a template upfront. A full worked trace (real prompt text at each step) is coming when `example.js` is built.

## The dev's job doesn't shrink — it shifts to building the machinery
As a dev you still build every piece precisely: what memory store exists, what tools are registered, what the executor loop's steps are, what counts as "done." The LLM's autonomy is scoped to choosing *which* of those you-built options to use and *when*, at runtime — never whether those options exist at all. You never hard-code "if the user says X, retrieve memory" — but you absolutely still build the retrieval call, wire it into the loop, and register every tool yourself.
