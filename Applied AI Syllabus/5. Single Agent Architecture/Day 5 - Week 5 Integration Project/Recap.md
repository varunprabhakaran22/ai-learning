# Day 5 Recap — Week 5 Integration Project

## Why this comparison, and how to read it
Every row below is a mechanism covered somewhere in Weeks 3-5. None of them are competitors to each other — they solve different, narrow problems, and a real agent (like `example.js`) uses several of them stacked together at once. The recurring trap this table exists to prevent: reaching for "add an agent" or "add RAG" when the actual gap is narrower (e.g. the model just needs one live number, not a whole planning loop). For each row: what problem it exists to solve, what it can't do by itself, and the concrete trigger for reaching for it.

## The comparison

| Mechanism | Problem it solves | What it does NOT solve | Reach for it when |
|---|---|---|---|
| **Plain LLM call** (Week 1) | Turn a structured prompt into structured text/JSON. Stateless — no memory of past calls, no ability to act on the world. | Can't fetch live/private data, can't take actions, can't do anything across more than one call. | The task is fully answerable from the prompt + model's training data alone, in one shot. |
| **Tool use / function calling** (Week 3) | Lets the model request a REAL action or REAL data (an API call, a DB read, a calculation) — your code executes it, injects the result back, model continues. This is the "Action" pillar — the model's only way to affect or observe the outside world. | Doesn't decide what to do with the result across multiple steps — that's still just one request→execute→inject→continue cycle. No memory of past sessions. No forward planning of multiple steps before acting. | The model needs a live fact it can't know from training (today's price, this user's account status) or needs to cause a real side effect (send an email, write a row). |
| **RAG** (Week 4) | The model's private/large knowledge problem: your documents aren't in its training data and won't fit in one prompt. Embed docs once, embed the query, cosine-similarity search, pull back only the topK relevant chunks, inject into the prompt. | Doesn't take actions (it's read-only retrieval, not a tool that changes anything). Doesn't decide *when* to search more, replan, or track a multi-step task. No sense of "this session" vs "last week's session." | The knowledge needed is large, private, or changes-too-often-to-retrain-on, but a normal tool call isn't the right shape for it (it's unstructured documents, not a structured API). |
| **Agent** (Week 5 Day 1, the umbrella) | An architecture where control flow — which tool, when, what next — lives in the model's own repeated reasoning at runtime, instead of in fixed code you wrote step-by-step (that fixed-code version is a "chain," not an agent). Four pillars: Tools, Planning, Memory, Executor. | Not a new model capability or a new API feature — every individual step inside it is still the same "structured prompt in, structured text out" call from Week 1. | The task's steps and their order genuinely can't be hard-coded in advance — what happens next depends on what a previous step's real result was. |
| **Planning** (Week 5 Day 3, one of the 4 pillars) | Before any tool-calling starts, one LLM call outputs an ordered subtask list with dependencies — your code then walks that dependency graph (plain topological sort, not a model decision) to pick what runs next. Turns "figure it out step by step live" into "decide the shape of the whole task upfront." | Doesn't execute anything itself (a plan is just a list until the executor runs it). Doesn't notice when it's gone stale — that's replanning's job, triggered by a code-level check after each result comes back, not a planning-time feature. | The task has multiple steps with real dependencies between them (step 3 needs step 1's output) and you want that structure decided once, not re-derived from scratch every turn. |
| **Memory** (Week 5 Day 2, one of the 4 pillars) | Fixes two things a short tool-use loop hits fast: the context window's hard token limit, and attention dilution from cramming in irrelevant history. Four types — in-context (current conversation), external/vector (RAG's exact mechanism, repointed at the agent's own past actions), episodic (summarized past sessions), semantic (durable standing facts). | Doesn't decide what to DO with retrieved memory — that's still a normal prompt-injection + model-reasoning step. Doesn't itself detect session boundaries (an app-level event has to trigger the save). | The task spans more history than fits in one context window, or needs to recall something from a genuinely separate past session, not just this conversation. |
| **Human-in-the-loop gating** (Week 5 Day 4, one of the 4 pillars) | Some tool calls are irreversible or costly if wrong — this inserts a pause between "model decided an action" and "code executes it," gated by a risk tier (always-ask / never-ask / threshold-based on confidence) assigned to the TOOL at registration time, never decided by the model at runtime. | Doesn't gate the LLM call itself (an LLM call alone has no real-world side effect — only a tool touching the outside world does). Doesn't replace replanning — a rejection just feeds back in as a new real fact, same path as any other broken assumption. | A registered tool can cause real, hard/impossible-to-undo consequences if the model calls it wrongly (send a real message, delete a record, spend real money). |
| **Executor loop** (Week 5 Day 4th pillar, the glue) | The runtime loop that actually ties Tools + Planning + Memory + Gating together: pull next eligible subtask → retrieve relevant memory → decide action → gate check → execute → detect broken assumptions → replan if needed → repeat until done. | Doesn't add a new mechanism of its own — every piece inside it is a mechanism from an earlier row. It's just the wiring, not a fifth capability. | You have more than one of the above pillars in play and need something to sequence them every iteration — this is what `example.js` actually is. |

## The one-line mental model for telling them apart
- **Plain call** = no state, no action, one shot.
- **Tools** = the model can act/observe the real world, one call at a time.
- **RAG** = the model can see YOUR large/private knowledge, read-only.
- **Memory** = the agent can see ITS OWN past (this session or an earlier one) without it consuming the whole context window.
- **Planning** = the shape of a multi-step task is decided upfront, not improvised turn by turn.
- **HITL gating** = a risky real-world action gets a pause before it fires, based on the tool's own risk tier.
- **Agent** = the umbrella architecture where all of the above are wired together and the model — not your fixed code — decides the control flow at runtime.

**From our discussion:** the four Week 5 pillars aren't four separate techniques competing for the same slot — Tools is the only pillar that touches the outside world at all; Planning and Memory are both just "structured prompt in, structured list/summary out" applied to a whole task and to past history respectively; HITL gating doesn't add a new capability, it inserts one pause point into the existing tool-execution step. The "agent" label just means these get chosen and sequenced by the model at runtime instead of by code you wrote in advance.
