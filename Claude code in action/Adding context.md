# Adding Context

Source: [Claude Code in Action — Adding context](https://anthropic.skilljar.com/claude-code-in-action/303241)

## Why context management matters
A project can have dozens or hundreds of files, but Claude only needs the relevant ones. Too much irrelevant context actually **decreases** Claude's performance — so the skill is guiding Claude toward the right files and docs, not dumping everything in.

## The `/init` command
Run `/init` when starting Claude in a new project. It tells Claude to analyze the entire codebase and understand:
- The project's purpose and architecture
- Important commands and critical files
- Coding patterns and structure

After analyzing, Claude writes a summary to `CLAUDE.md`. When Claude asks permission to create the file, either:
- Hit **Enter** to approve each write, or
- Press **Shift+Tab** to let Claude write freely for the rest of the session.

## The `CLAUDE.md` file
Two main purposes:
1. Guides Claude through the codebase — commands, architecture, coding style.
2. Lets you give Claude custom/specific directions.

This file is included in **every request** — effectively a persistent system prompt for the project.

## CLAUDE.md file locations
| File | Scope |
|---|---|
| `CLAUDE.md` | Generated by `/init`, committed to source control, shared with the team |
| `CLAUDE.local.md` | Not shared — personal instructions/customizations |
| `~/.claude/CLAUDE.md` | Applies to **all** projects on your machine |

## Adding custom instructions
Edit `CLAUDE.md` directly, or run `/memory` inside Claude Code to open it. Example: if Claude over-comments code, add an instruction like:

> Use comments sparingly. Only comment complex code.

Claude reads `CLAUDE.md` at the start of every conversation, so edits apply starting with your **next** message.

## File mentions with `@`
Use `@` followed by a file path to include that file's contents in your request.

Example:
```
How does the auth system work? @auth
```
Claude shows a list of matching files to pick from, then includes the selected file's contents.

## Referencing files inside CLAUDE.md
The same `@` syntax works inside `CLAUDE.md` — useful for files relevant across many parts of the project (e.g. a DB schema).

Example:
```
The database schema is defined in the @prisma/schema.prisma file. Reference it anytime you need to understand the structure of data stored in the database.
```
Mentioning a file this way means its contents are pulled into **every request automatically** — Claude doesn't need to search for and read it each time.

### Interop with AGENTS.md
If a repo already has an `AGENTS.md` for another tool, don't duplicate instructions. Instead, add `@AGENTS.md` as the **first line** of `CLAUDE.md` — Claude loads that file's contents first, then reads any Claude-specific instructions you add below the import.

## Key takeaway
- Run `/init` in a new project so Claude writes a `CLAUDE.md` summary of purpose, architecture, and conventions.
- `CLAUDE.md` is a persistent system prompt, re-read at the start of every conversation — edit it directly or via `/memory`.
- Three scopes: `CLAUDE.md` (shared/committed), `CLAUDE.local.md` (personal, not shared), `~/.claude/CLAUDE.md` (all projects on your machine).
- `@path` pulls a file's contents into a request on demand; the same syntax inside `CLAUDE.md` pulls it into *every* request automatically.
- If `AGENTS.md` already exists, import it with `@AGENTS.md` as the first line of `CLAUDE.md` instead of duplicating instructions.
